# awl v0.6.x 구현 사양 및 하네스 무게 최종 검증 보고서 (개정본)

이 보고서는 `docs/codex-prompt.md`에 작성된 6대 구현 사양이 실제 `agent-work-loop` (awl) 코드베이스의 아키텍처와 충돌하지 않는지, 그리고 구현 시 하네스 자체가 지나치게 복잡해지거나 무거워질 우려가 없는지 분석한 기술 검증서입니다.

---

## 1. 종합 평가 요약

> [!TIP]
> **검증 요약: "대체로 매우 실용적이며 가벼우나, Git Hooks 자동 등록 부분만 주의 요망"**
> - **매우 가벼움 (4개)**: `raw_request` 보존, 3회 실패 시 `blocked` 전이, `protectedFiles` 감시, Gate 1 진입 가드. 이들은 현 코드베이스에 이미 존재하는 헬퍼 함수들을 재사용하므로 코드 추가가 10~20줄 내외로 극도로 가볍습니다.
> - **적절함 (1개)**: `Changelog` 초안 자동 제안 기능은 기존 `records` 수집 파서를 재사용하므로 안전합니다.
> - **잠재적 위험 (1개 - 무거워질 우려)**: **`npm publish` 및 `git push` 물리적 차단 훅 자동 등록**은 Windows와 macOS 간의 권한(`chmod +x`) 및 실행 셸 환경 차이로 인해 예외 코드가 늘어나 아키텍처가 비대해질 위험이 있습니다.

## 2. 요구사항별 정밀 검증 및 잠재적 설계 결함 분석

### [1] 세션 복구를 위한 `raw_request` 보존
* **방향성**: **매우 적절함 (초경량)**
* **codebase 대조**: 
  * `src/commands/state.ts`의 `loadState`와 `writeState`는 JSON을 단순 직렬화합니다. `state.json`에 `raw_request: string | null` 필드를 추가하는 것은 상태 파일의 스키마 변경 수준이므로 의존성이나 결합도가 전혀 늘어나지 않습니다.
* **잠재적 결함 발견**:
  * `work.ts`에서 `awl work switch` 등으로 워크아이템을 전환할 때 호출되는 `archiveCurrent` 함수 내의 구조분해 할당(L161-175) 리스트에서 `raw_request`를 명시적으로 비워주고, 아카이브 스냅샷(`WorkitemEntry`)과 복원 로직(`restoreWorkitem`)에 추가해 주지 않으면 다른 워크아이템 전환 시 요구사항 원문이 날아가거나 꼬이는 버그가 발생합니다. 구현 지침에 이 매핑 가이드를 추가해야 합니다.

### [2] 3회 검증 실패 시 기계적 `blocked` 전이
* **방향성**: **매우 적절함 (기존 헬퍼 재사용)**
* **codebase 대조**:
  * `src/commands/state.ts` 내부의 `setCriterion` 함수가 이미 `criteria` 배열 내의 특정 ID 상태를 찾아 부분 병합하는 안정적인 계약을 구현하고 있습니다.
* **잠재적 결함 발견**:
  * 검증 실패 시마다 무조건 `attempts`를 증가시키기만 하고 검증 통과(Success) 시 리셋해주지 않으면, 개발 과정 중 수정과 빌드 통과를 정상적으로 반복하다가 누적으로 3회 실패에 도달했을 때 불필요하게 `blocked`로 영구 중단되는 모순이 발생합니다. **검증 성공 시 `attempts` 카운트를 0으로 리셋**하는 예외 규칙이 필수입니다.

### [3] 설정 기반 보호 파일(`protectedFiles`) 무단 수정 방지
* **방향성**: **매우 적절함 (기존 코드 재사용)**
* **codebase 대조**:
  * 현재 `src/commands/doctor.ts` L106에 `gitDirtyFiles(projectRoot)`가 이미 완벽하게 구현되어 동작 중입니다.
* **잠재적 결함 발견**:
  * 사람이 직접 의도적으로 설정을 바꾸고자 할 때도 하네스가 완전히 차단하면 변경이 불가능한 데드락 상태에 빠집니다. 인간 개발자의 직접 제어를 위해 차단 검사를 일시 우회할 수 있는 **`--force` 옵션**이 제공되어야 합니다.

### [4] Gate 1 승인 전 코드 수정성 커맨드 차단
* **방향성**: **매우 적절함 (기존 phase 상태 재사용)**
* **codebase 대조**:
  * `state.ts` L160의 `runStateSet`에 이미 `phase === 'loop'` 시 게이트 유효성을 검사하는 `requireGateForLoop` 훅 프로토콜이 존재합니다.
* **잠재적 결함 발견**:
  * `awaiting-gate1` 상태에서 모든 수정 명령을 차단해 버리면, 정작 계획을 기입하고 승인을 요청해야 하는 `awl record gate` 등 계획 수립 관련 필수 명령마저 차단되어 승인 절차를 시작할 수 없는 데드락이 발생합니다. `record gate`, `work switch`, `work abandon` 등 계획 준비용 명령들은 차단 대상 예외(Allowlist)로 분류해야 합니다.

### [5] 완료 시 Changelog 초안 자동 제안 기능
* **방향성**: **적절함 (기존 records 리더 활용)**
* **codebase 대조**:
  * `src/commands/evolve.ts`가 이미 `.awl/records/` 하위의 json 기록들을 취합해 gotcha를 분석하고 측정 지표를 계산하고 있습니다.
  * 이 파일 취합 논리를 살짝 재사용하여 각 record의 `what`, `why`, `lesson`을 단순 마크다운 문자열 템플릿으로 출력해주는 보조 명령어를 얹는 수준이므로 결합도가 매우 낮습니다.

## 3. `git/hooks` 물리적 훅 등록의 구조적 결함 및 경량화 방안

* **물리적 차단 결함**:
  * `pre-push` 훅은 오직 `git push` 실행 시에만 트리거되므로, 사용자가 터미널에 `npm publish`를 입력할 때는 이 훅이 절대 실행되지 않아 배포를 막을 수 없습니다.
  * 따라서 `npm publish` 차단은 `package.json`의 `prepublishOnly` 스크립트에 차단 커맨드를 지정하거나, `npm/pnpm/yarn` 배포 자체를 방어하는 복합 장치로 가이드라인을 다듬어야 합니다.

* **실용적인 경량화 대안**:
  * **방안 A: 훅 템플릿 배포 최소화**
    * 복잡한 동적 훅 관리를 CLI 내부 코드로 짜지 않고, 단순 예시 파일(쉘 스크립트 한 블록)을 `engineDir` 아래의 템플릿(예: `engine/templates/pre-push.sample`)으로 두고 `awl init` 시 복사만 해주도록 극도로 절차를 고정합니다. 권한 설정 실패 등은 경고(Warning)만 하고 크래시되지 않게 예외를 철저히 삼킵니다.
  * **방안 B: awl CLI 수준의 래핑 검사 (강력 추천)**
    * 굳이 Git 훅을 건드리지 않더라도, 에이전트 스킬(`awl-loop`) 내에서 실행되는 유일한 도구가 `awl`이므로, `awl` 내부에서 `cross-spawn`으로 임의의 사용자 스크립트를 빌드/배포하기 전(`awl release` 등의 명령이 추가될 경우) 내부 실행 가드 스위치를 통하는 방식을 씁니다. (현재 구조에서는 방안 A가 가장 가볍고 제안 사양과 부합합니다.)
